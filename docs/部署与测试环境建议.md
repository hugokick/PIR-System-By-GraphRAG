# 部署与测试环境建议

> 目的：把项目源码放到 GitHub 做版本管理，并提供一个稳定的云端测试入口，便于后续开发、验收和演示。

## 推荐结论

建议采用“两层环境”：

1. **GitHub 存源码**
   - 负责代码版本管理、协作、Issue、变更记录。
   - 每次开发完成后提交并推送。

2. **云端测试环境跑系统**
   - 当前纯前端 Demo 可先用 GitHub Pages、Vercel 或 Netlify。
   - 后续接入 FastAPI/NestJS、PostgreSQL、MinIO 后，建议迁移到云服务器 + Docker Compose。

## 方案 A：GitHub + GitHub Pages

适合当前阶段：前端 Demo、无需后端、快速演示。

优点：

- 成本低
- 配置简单
- 每次 push 后自动构建
- 适合给客户或团队发链接测试

限制：

- 只适合静态前端
- 无法直接运行 PostgreSQL、MinIO、后端 API
- 当前 localStorage 数据只保存在访问者浏览器中

本项目已预留 GitHub Pages 工作流：

```text
.github/workflows/deploy-pages.yml
```

推送到 GitHub 后，在仓库设置中启用 Pages，并选择 GitHub Actions 作为部署来源即可。

## 方案 B：GitHub + Vercel/Netlify

适合当前阶段：前端 Demo 和高频预览。

优点：

- 比 GitHub Pages 更适合前端预览
- 每个分支或 PR 可以生成预览链接
- 配置简单

限制：

- 如果后续有复杂后端、数据库、文件服务，仍建议转云服务器

## 方案 C：GitHub + 云服务器 + Docker Compose

适合下一阶段：真实 MVP 试用环境。

建议组件：

- 前端：React/Vite 构建后的静态文件，由 Nginx 托管
- 后端：FastAPI 或 NestJS
- 数据库：PostgreSQL
- 向量检索：pgvector
- 文件存储：MinIO
- 反向代理：Nginx

优点：

- 最接近真实生产环境
- 能测试后端、数据库、文件上传、RAG、权限
- 后续可平滑演进为正式部署

限制：

- 需要服务器和域名配置
- 需要维护部署脚本、备份和安全策略

## 建议阶段路线

### 第 1 步：先上 GitHub

目标：建立源码版本管理。

操作：

```bash
git add .
git commit -m "feat: initialize exhibit atlas mvp"
git branch -M main
git remote add origin https://github.com/<your-org-or-user>/<repo-name>.git
git push -u origin main
```

注意：`<your-org-or-user>` 和 `<repo-name>` 需要替换为实际 GitHub 用户/组织和仓库名。

### 第 2 步：启用 GitHub Pages

目标：给当前前端 Demo 一个公开测试地址。

操作：

1. 打开 GitHub 仓库
2. 进入 `Settings -> Pages`
3. Source 选择 `GitHub Actions`
4. 推送代码后等待 Actions 完成
5. 使用 Pages 地址进行演示和测试

### 第 3 步：后端开发后切到云服务器

目标：支持真实数据库、文件上传和智能检索。

推荐部署方式：

```text
云服务器
├── nginx
├── frontend
├── backend
├── postgres + pgvector
└── minio
```

后续应补充：

- `backend/Dockerfile`
- `deploy/frontend.Dockerfile`
- `docker-compose.cloud.yml`
- `.env.example`
- 数据库迁移脚本
- 部署说明

当前测试环境已经增加云服务器 Docker Compose 配置：

```bash
docker compose -f docker-compose.cloud.yml up -d --build
```

默认公网测试端口：

```text
http://<服务器公网 IP>:18080/
```

如果服务器安全组不开放高端口，可由宿主机 Nginx 挂载到子路径：

```text
http://<服务器公网 IP>/pir-system/
```

该入口由 Nginx 托管前端静态文件，并把 `/api/*` 与 `/health` 反向代理到 FastAPI 后端。当前云端构建参数为 `VITE_API_BASE_URL=/pir-system`，用于让子路径部署下的前端请求 `/pir-system/api/*`，避免撞到宿主机已有系统的 `/api/*`。

云端 Compose 已包含内部 PostgreSQL 服务：

- 服务名：`postgres`
- 数据库：`exhibit_atlas`
- 数据卷：`postgres_data`
- 后端通过 `DATABASE_URL` 自动启用 PostgreSQL 仓库

PostgreSQL 不映射宿主机端口，避免和服务器上已有数据库服务冲突。

## 当前项目是否适合上传 GitHub

适合。当前项目已经具备：

- `.gitignore`
- `package.json`
- `package-lock.json`
- `README.md`
- `docs/展项图鉴查询MVP系统开发计划.md`
- 可执行测试命令
- 可执行构建命令

不应上传：

- `node_modules/`
- `dist/`
- `.env`
- 本地临时文件

这些已经写入 `.gitignore`。

## 推荐选择

当前立即推荐：

> **GitHub 私有仓库 + GitHub Pages 或 Vercel 测试地址**

等后端和数据库开始开发后，再升级为：

> **GitHub + 云服务器 Docker Compose 测试环境**
