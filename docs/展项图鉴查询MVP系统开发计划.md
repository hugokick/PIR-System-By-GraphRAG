# 展项图鉴查询 MVP 系统开发计划

> 版本：v0.1  
> 日期：2026-06-30  
> 项目定位：面向科技馆、博物馆等展陈场景的展项数字档案、结构化检索、轻量知识图谱与智能问答 MVP。

## 1. 项目目标

本项目面向展陈行业，帮助科展人员沉淀公司过往展项展品资料，支持按造价、材质、主题、场馆类型、互动形式、项目案例等维度快速检索，并通过图谱关系展示“展项-项目-业主-材料-供应商-主题-相似展项”的关联。

MVP 阶段目标不是建设完整自动知识图谱平台，而是完成一套可演示、可试用、能支撑业务验证的系统：

- 展项数字档案
- 结构化筛选
- 轻量知识图谱构建与展示
- 语义检索雏形
- 基于资料来源的智能问答雏形
- 基础管理后台

## 2. 当前项目状态

当前仓库已经完成 React + Vite + TypeScript 前端 Demo：

- 展项档案列表、详情、新增录入
- 场馆类型、类别、主题、材料、交互方式、状态、关键词筛选
- 轻量中文语义检索雏形
- 前端图谱展示
- 图片/文档上传入口
- 基础数据看板

当前数据仍主要来自前端种子数据和浏览器 localStorage。下一步应推进为真实后端、数据库和可持续构建的知识图谱数据模型。

## 3. MVP 核心范围

### 3.1 展项建档

字段包括：

- 展项名称
- 类别
- 主题
- 适用场馆
- 造价区间
- 材质
- 尺寸
- 交互方式
- 供应商
- 项目年份
- 业主
- 状态
- 标签
- 展项说明

### 3.2 多媒体档案

支持上传和关联：

- 图片
- 视频
- 设计图
- 报价文件
- 说明文档
- Excel 资料

### 3.3 分类与结构化检索

支持按以下维度筛选：

- 展馆类型
- 学科领域
- 主题
- 预算区间
- 材质
- 互动形式
- 项目案例
- 状态

### 3.4 轻量知识图谱

图谱构建是本项目的核心主线之一，不应仅作为前端可视化组件处理。

MVP 阶段采用 PostgreSQL 关系表表达图谱，不优先引入 Neo4j。通过实体表和关系表沉淀图谱结构，再由后端提供图谱查询 API 给前端展示。

核心实体：

- 展项 `Exhibit`
- 项目 `Project`
- 业主 `Owner`
- 供应商 `Supplier`
- 材料 `Material`
- 主题 `Theme`
- 交互方式 `Interaction`
- 媒体资产 `MediaAsset`
- 文档资料 `Document`
- 展项关系 `ExhibitRelation`

核心关系：

- 展项 -> 项目
- 展项 -> 业主
- 展项 -> 供应商
- 展项 -> 材料
- 展项 -> 主题
- 展项 -> 交互方式
- 展项 -> 媒体/文档
- 展项 -> 相似展项

### 3.5 智能检索与问答

MVP 中先实现：

- 关键词检索
- 结构化过滤
- 向量语义检索
- 混合排序
- 匹配原因说明

后续在资料积累后接入 RAG 问答，要求回答必须带来源引用。

## 4. 推荐技术架构

### 4.1 前端

- React + TypeScript
- Vite 或 Next.js
- 图谱可视化可选 ECharts、React Flow、Cytoscape
- UI 可继续保持当前轻量工作台风格，后续再决定是否接 Ant Design 或 shadcn/ui

### 4.2 后端

推荐二选一：

- FastAPI：适合 Python AI/RAG、文档解析、embedding 流程
- NestJS：适合企业后台、模块化权限和长期工程化

若后续重点是 AI 检索与文档处理，优先推荐 FastAPI。

### 4.3 数据库与存储

- PostgreSQL：主数据库
- pgvector：向量检索
- MinIO 或云对象存储：媒体和文档文件
- PostgreSQL 关系表：MVP 阶段表达轻量知识图谱

### 4.4 AI 能力

- embedding：对展项描述、标签、材料、交互方式、文档切片生成向量
- 混合检索：结构化过滤 + 关键词 + 向量召回
- RAG 问答：基于展项档案和上传资料回答，并返回引用来源

## 5. 阶段计划

### 阶段 0：固化当前 Demo

目标：把现有前端演示版变成可继续迭代的稳定基线。

交付：

- 提交当前代码
- 整理 README
- 明确字段说明和样例数据
- 保留现有前端 Demo 作为 v0.1 基线

验收：

- `npm test` 通过
- `npm run build` 通过
- 页面可正常展示列表、详情、筛选、图谱和新增表单

### 阶段 1：领域模型与图谱数据模型

目标：把展项档案和知识图谱关系建模清楚。

交付：

- 数据库 ER 模型
- 图谱实体与关系说明
- 表结构设计
- API 草案

建议表：

- `exhibits`
- `projects`
- `owners`
- `suppliers`
- `materials`
- `themes`
- `interactions`
- `media_assets`
- `documents`
- `exhibit_materials`
- `exhibit_interactions`
- `exhibit_relations`
- `exhibit_documents`

验收：

- 能从数据模型明确表达“展项-项目-业主-供应商-材料-主题-相似展项”的关系
- 后端可根据关系表生成图谱 nodes 和 edges

### 阶段 2：后端与 PostgreSQL

目标：把 localStorage Demo 升级为真实可试用系统。

交付：

- 后端项目初始化
- PostgreSQL 连接
- 数据库迁移
- 展项 CRUD API
- 筛选查询 API
- 展项详情 API
- 图谱查询 API

关键 API：

```http
GET /api/exhibits
POST /api/exhibits
GET /api/exhibits/{id}
PUT /api/exhibits/{id}
DELETE /api/exhibits/{id}
GET /api/exhibits/{id}/graph
```

图谱 API 返回示例：

```json
{
  "nodes": [
    { "id": "exhibit:1", "label": "杠杆乐园", "type": "exhibit" },
    { "id": "material:metal", "label": "金属", "type": "material" }
  ],
  "edges": [
    { "source": "exhibit:1", "target": "material:metal", "label": "使用材料" }
  ]
}
```

验收：

- 前端列表、详情、筛选全部走后端接口
- 图谱数据由后端根据关系表生成
- 数据刷新后仍可保留

### 阶段 3：图谱构建流程

目标：让系统能持续构建知识图谱，而不是只展示静态关系。

交付：

- 新增展项时同步生成实体和关系
- 编辑展项时同步更新关系
- 支持手动维护相似展项
- 支持从 Excel 导入时生成材料、主题、供应商、业主、项目关系
- 支持图谱关系编辑

验收：

- 新增一个展项后，能自动生成图谱节点和边
- 修改材料、主题、供应商后，图谱同步更新
- Excel 导入后可直接查看图谱关系

当前进展：

- PostgreSQL 运行时 schema 已补齐 `owners` / `projects` / `suppliers` / `themes` / `materials` / `interactions` / `exhibits` / 关系表等标准实体模型，并在展项新增、编辑、删除、导入后从当前有效档案重建实体关系投影
- 前端知识图谱窗口已接入 NVL，并保留当前展项子图 / 全库 Neo4j 演示图谱模式切换
- 图谱面板已显示数据源、节点数、关系数、关系 type、节点详情，并支持 force-directed 布局、缩放、平移、拖拽和重新布局
- 图谱画布已缩小节点、拉开初始关系距离，并增加节点 label / 关系 type 的只读可视标签，避免画布下方出现大块无效占位
- 普通新增 / 编辑展项接口已校验 `related_exhibit_ids`，拒绝不存在展项或自引用关系，避免从手工录入口写入无效图谱关系
- PostgreSQL 运行时仓储已新增 `kg_nodes` / `kg_edges` 投影表，新增、编辑、删除展项后会从当前有效展项重建 KG 快照并同步节点/边，用于 GraphRAG 上下文和关系表未命中时的兼容回退
- `GET /api/exhibits/{id}/graph` 已优先读取 PostgreSQL 标准实体 / 关系表生成当前展项一跳子图；关系表没有命中时继续回退到 KG 投影表，查询演示库专有展项时继续回退到 Neo4j 演示图谱服务
- 媒体资产已作为 `media_asset` 节点和 `has_media` 关系进入标准关系表图谱 API 与 KG 快照，和文档资料的 `has_document` 关系保持一致
- 当前展项图谱的一跳子图已同时包含出边与入边的相似展项关系，避免只在关系发起方查看时才能看到 `similar_to`
- GraphRAG 检索 / 问答接口已在 PostgreSQL 仓储可用时复用 `kg_nodes` / `kg_edges` 投影快照，避免每次请求临时重建图谱上下文

### 阶段 4：文件与多媒体档案

目标：让展项档案真正承载项目资料。

交付：

- 文件上传接口
- 图片/PDF 预览
- Word、Excel、报价文件下载
- 文件与展项绑定
- MinIO 或本地对象存储
- 文件元数据入库

验收：

- 每个展项可绑定多份资料
- 文件可预览或下载
- 文件资料能作为后续 RAG 的来源
- 前端上传入口应覆盖图片、视频、PDF、Office、Excel/CSV 与文本类资料；文本类资料按文档资料入库并进入 RAG 引用来源链路

当前进展：

- 上传的文本 / PDF 资料已进入文档切片与 GraphRAG 引用来源链路
- 上传接口会拒绝未知资料类型，避免错误的 `asset_kind` 被静默当作媒体入库
- 管理员可在展项详情中删除误传的媒体或资料，删除后刷新展项资料列表
- 删除上传资料时同步清理本地对象存储文件，旧文件 URL 会失效

### 阶段 5：Excel 导入

目标：降低历史资料录入成本。

交付：

- Excel 模板
- 上传 Excel
- 字段映射
- 导入预览
- 错误行提示
- 批量写入展项与图谱关系

验收：

- 可通过 Excel 一次导入多条展项
- 材料、主题、供应商、业主等关系能自动入库
- 导入失败时能指出具体行和字段
- 相似展项字段必须引用已有展项或同批导入展项，避免生成无效图谱关系

当前进展：

- 已支持 CSV / XLSX 导入预览、提交写入和 Excel 模板下载
- CSV 导入已兼容 UTF-8 / UTF-8 BOM / GB18030 编码，便于接收中文历史资料表
- 损坏或无法解析的 XLSX 文件会返回结构化导入文件错误，避免导入预览出现服务端 500
- 空导入文件会返回结构化错误提示，避免业务人员看到“0 行 0 错误”而无法判断问题
- 前端导入入口已显示后端结构化解析错误中的文件名与支持格式，便于业务人员快速修正文件
- 前端导入预览会把后端字段名和常见校验错误转换为中文业务提示，减少历史资料整理时的理解成本
- 导入时会校验必填字段、整数预算、预算上下限和相似展项引用有效性
- 同一个导入文件内出现重复展项编号时会标记所有重复行并拒绝入库，避免同批数据互相覆盖
- 同批导入的展项可互相引用为相似展项，提交后可在图谱中查看 `similar_to` 关系
- 导入提交会写入逐条创建/更新审计日志，并记录批量导入摘要，便于追踪历史资料入库

### 阶段 6：混合检索

目标：把当前规则语义检索升级为真实检索能力。

交付：

- pgvector 扩展
- 展项 embedding
- 文档切片 embedding
- 结构化过滤 + 关键词 + 向量检索
- 混合排序
- 匹配原因说明

示例查询：

> 找几个适合低龄儿童、预算不高、互动性强的力学展项

期望返回：

- 杠杆乐园：匹配低龄儿童、力学、机械互动、预算 20-35 万
- 滑轮挑战墙：匹配低预算、力学、多人协作、机械互动

验收：

- 检索结果能结合结构化条件和语义相似度
- 每条结果能说明主要匹配原因

当前进展：

- 混合检索已支持结构化过滤、关键词匹配与 pgvector 语义分数召回
- 已新增规则 fallback 版查询理解模块，能从中文自然语言中识别主题、人群、预算倾向、材料、互动、标签、参照案例和排除项，并作为混合检索与 GraphRAG 的 query rewrite / rerank 信号；当前不调用外部 LLM，保持 API 契约不变
- 标签已作为独立结构化筛选条件贯通列表、看板、混合检索和 GraphRAG 问答过滤，不再只能依赖关键词模糊匹配
- 业主、供应商已作为独立结构化筛选条件贯通列表、看板、混合检索和 GraphRAG 问答过滤，便于按项目服务对象和实施单位精确查档
- GraphRAG 问答过滤已补齐类别与项目案例维度，和前端结构化筛选栏、普通列表、混合检索保持一致
- 查询提到“预算不高 / 低预算”等自然语言意图时，低预算展项会获得排序加权，并返回具体预算区间作为匹配原因
- 查询提到“不要 / 排除 / 不考虑”等排除意图时，混合检索与 GraphRAG 检索会把命中排除条件的候选展项过滤掉，而不是仅做轻微降权
- 上传文档生成的文本切片已参与普通混合检索；命中文档资料时，结果会返回对应资料名称作为匹配原因

### 阶段 7：RAG 智能问答

目标：基于库内资料回答问题，并返回引用来源。

交付：

- 文档解析
- 文档切片
- 向量召回
- 问答接口
- 引用来源返回
- 不确定性处理

验收：

- 问答必须基于展项档案或上传资料
- 回答中包含来源引用
- 资料中没有依据时，明确说明未找到依据

当前进展：

- GraphRAG 回答已返回编号引用来源
- 上传资料命中时，回答的来源说明会包含具体文档片段，便于人工核验依据
- GraphRAG 检索命中文档资料时，匹配原因会返回“匹配资料：资料名称”，避免把内部字段名暴露给业务人员
- GraphRAG 检索 / 问答已支持从仓储读取 KG 投影快照，同时保留展项档案和上传资料的引用证据链路
- 前端 GraphRAG 文档引用卡已支持直接打开原始资料和下载资料，同时保留点击引用定位到所属展项，便于业务人员核验答案依据
- GraphRAG 回答命中候选展项但缺少引用来源时，前端会明确显示“未找到可引用来源”提示，不把候选列表误呈现为已可采信答案

### 阶段 8：管理后台与权限

目标：支持业务人员安全维护数据。

交付：

- 登录
- 角色权限
- 管理员、编辑、只读访客
- 审核状态
- 操作日志
- 删除保护

验收：

- 不同角色能访问不同功能
- 数据变更有记录
- 重要资料不能被误删

当前进展：

- 已具备管理员、编辑、访客角色控制
- 前端已接入演示账号登录，登录后使用 Bearer token 调用受保护接口，并在刷新后恢复会话
- 已提供当前用户校验接口；前端恢复本地会话时会校验 token，失效时自动回到登录状态
- 资料删除仅允许管理员执行，编辑 / 访客不可删除
- 资料删除会写入操作日志，便于追踪误删风险
- 编辑角色不能通过新增 / 编辑展项接口自行写入“已审核”等审核状态，普通编辑改动会进入或保持既有审核流程
- 前端编辑表单已对非管理员隐藏审核状态字段，审核通过 / 退回统一走管理员详情页专用操作
- 操作日志前端已覆盖新增、编辑、删除、审核、上传/删除资料、相似展项关系维护和批量导入摘要等主要动作的可读标签

## 6. 近期推荐开发顺序

最近 1-2 周建议优先做：

1. 提交当前前端 Demo
2. 设计 PostgreSQL 图谱数据模型
3. 搭建后端 API
4. 前端接真实接口
5. 实现图谱查询 API
6. 实现新增/编辑展项时的关系表同步

这条路线最关键，因为它把系统从“前端演示”推进到“真实数据驱动的 MVP”。

## 7. 暂不纳入 MVP 的内容

以下内容建议放到二期或更后面：

- 全自动知识图谱构建
- CAD/3D 深度解析
- 精准造价预测
- 复杂图数据库推理
- 多智能体自动生成展陈方案
- ERP/OA/合同系统深度集成
- 大规模审批流

## 8. 成功标准

MVP 成功标准：

- 业务人员能录入真实展项
- 能按结构化条件快速筛选
- 能查看展项详情和资料
- 能看到展项与项目、业主、供应商、材料、主题、相似展项的图谱关系
- 能通过自然语言检索得到可解释结果
- 能基于库内资料进行初步问答，并返回引用来源

一句话总结：

> 展项档案是数据入口，关系表是图谱基础，图谱 API 是展示和检索基础，RAG 是最后的智能层。
