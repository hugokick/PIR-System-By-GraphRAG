# 展项图鉴查询 MVP 系统开发计划

> 版本：v0.1  
> 日期：2026-06-30  
> 项目定位：面向科技馆、博物馆等展陈场景的展项数字档案、结构化检索、轻量知识图谱与智能问答 MVP。

## 1. 项目目标

本项目面向展陈行业，帮助科展人员沉淀公司过往展项展品资料，支持按造价、材质、主题、场馆类型、互动形式、项目案例等维度快速检索，并通过图谱关系展示“展项-项目-业主-材料-供应商-主题-相似展项”的关联。

MVP 阶段目标不是建设完整自动知识图谱平台，而是完成一套可演示、可试用、能支撑业务验证的系统：

- 展项数字档案
- 结构化筛选
- 轻量知识图谱构建与展示
- 语义检索雏形
- 基于资料来源的智能问答雏形
- 基础管理后台

## 2. 当前项目状态

当前仓库已经完成 React + Vite + TypeScript 前端 Demo：

- 展项档案列表、详情、新增录入
- 场馆类型、类别、主题、材料、交互方式、状态、关键词筛选
- 轻量中文语义检索雏形
- 前端图谱展示
- 图片/文档上传入口
- 基础数据看板

当前数据仍主要来自前端种子数据和浏览器 localStorage。下一步应推进为真实后端、数据库和可持续构建的知识图谱数据模型。

## 3. MVP 核心范围

### 3.1 展项建档

字段包括：

- 展项名称
- 类别
- 主题
- 适用场馆
- 造价区间
- 材质
- 尺寸
- 交互方式
- 供应商
- 项目年份
- 业主
- 状态
- 标签
- 展项说明

### 3.2 多媒体档案

支持上传和关联：

- 图片
- 视频
- 设计图
- 报价文件
- 说明文档
- Excel 资料

### 3.3 分类与结构化检索

支持按以下维度筛选：

- 展馆类型
- 学科领域
- 主题
- 预算区间
- 材质
- 互动形式
- 项目案例
- 状态

### 3.4 轻量知识图谱

图谱构建是本项目的核心主线之一，不应仅作为前端可视化组件处理。

MVP 阶段采用 PostgreSQL 关系表表达图谱，不优先引入 Neo4j。通过实体表和关系表沉淀图谱结构，再由后端提供图谱查询 API 给前端展示。

核心实体：

- 展项 `Exhibit`
- 项目 `Project`
- 业主 `Owner`
- 供应商 `Supplier`
- 材料 `Material`
- 主题 `Theme`
- 交互方式 `Interaction`
- 媒体资产 `MediaAsset`
- 文档资料 `Document`
- 展项关系 `ExhibitRelation`

核心关系：

- 展项 -> 项目
- 展项 -> 业主
- 展项 -> 供应商
- 展项 -> 材料
- 展项 -> 主题
- 展项 -> 交互方式
- 展项 -> 媒体/文档
- 展项 -> 相似展项

### 3.5 智能检索与问答

MVP 中先实现：

- 关键词检索
- 结构化过滤
- 向量语义检索
- 混合排序
- 匹配原因说明

后续在资料积累后接入 RAG 问答，要求回答必须带来源引用。

## 4. 推荐技术架构

### 4.1 前端

- React + TypeScript
- Vite 或 Next.js
- 图谱可视化可选 ECharts、React Flow、Cytoscape
- UI 可继续保持当前轻量工作台风格，后续再决定是否接 Ant Design 或 shadcn/ui

### 4.2 后端

推荐二选一：

- FastAPI：适合 Python AI/RAG、文档解析、embedding 流程
- NestJS：适合企业后台、模块化权限和长期工程化

若后续重点是 AI 检索与文档处理，优先推荐 FastAPI。

### 4.3 数据库与存储

- PostgreSQL：主数据库
- pgvector：向量检索
- MinIO 或云对象存储：媒体和文档文件
- PostgreSQL 关系表：MVP 阶段表达轻量知识图谱

### 4.4 AI 能力

- embedding：对展项描述、标签、材料、交互方式、文档切片生成向量
- 混合检索：结构化过滤 + 关键词 + 向量召回
- RAG 问答：基于展项档案和上传资料回答，并返回引用来源

## 5. 阶段计划

### 阶段 0：固化当前 Demo

目标：把现有前端演示版变成可继续迭代的稳定基线。

交付：

- 提交当前代码
- 整理 README
- 明确字段说明和样例数据
- 保留现有前端 Demo 作为 v0.1 基线

验收：

- `npm test` 通过
- `npm run build` 通过
- 页面可正常展示列表、详情、筛选、图谱和新增表单

### 阶段 1：领域模型与图谱数据模型

目标：把展项档案和知识图谱关系建模清楚。

交付：

- 数据库 ER 模型
- 图谱实体与关系说明
- 表结构设计
- API 草案

建议表：

- `exhibits`
- `projects`
- `owners`
- `suppliers`
- `materials`
- `themes`
- `interactions`
- `media_assets`
- `documents`
- `exhibit_materials`
- `exhibit_interactions`
- `exhibit_relations`
- `exhibit_documents`

验收：

- 能从数据模型明确表达“展项-项目-业主-供应商-材料-主题-相似展项”的关系
- 后端可根据关系表生成图谱 nodes 和 edges

### 阶段 2：后端与 PostgreSQL

目标：把 localStorage Demo 升级为真实可试用系统。

交付：

- 后端项目初始化
- PostgreSQL 连接
- 数据库迁移
- 展项 CRUD API
- 筛选查询 API
- 展项详情 API
- 图谱查询 API

关键 API：

```http
GET /api/exhibits
POST /api/exhibits
GET /api/exhibits/{id}
PUT /api/exhibits/{id}
DELETE /api/exhibits/{id}
GET /api/exhibits/{id}/graph
```

图谱 API 返回示例：

```json
{
  "nodes": [
    { "id": "exhibit:1", "label": "杠杆乐园", "type": "exhibit" },
    { "id": "material:metal", "label": "金属", "type": "material" }
  ],
  "edges": [
    { "source": "exhibit:1", "target": "material:metal", "label": "使用材料" }
  ]
}
```

验收：

- 前端列表、详情、筛选全部走后端接口
- 图谱数据由后端根据关系表生成
- 数据刷新后仍可保留

### 阶段 3：图谱构建流程

目标：让系统能持续构建知识图谱，而不是只展示静态关系。

交付：

- 新增展项时同步生成实体和关系
- 编辑展项时同步更新关系
- 支持手动维护相似展项
- 支持从 Excel 导入时生成材料、主题、供应商、业主、项目关系
- 支持图谱关系编辑

验收：

- 新增一个展项后，能自动生成图谱节点和边
- 修改材料、主题、供应商后，图谱同步更新
- Excel 导入后可直接查看图谱关系

### 阶段 4：文件与多媒体档案

目标：让展项档案真正承载项目资料。

交付：

- 文件上传接口
- 图片/PDF 预览
- Word、Excel、报价文件下载
- 文件与展项绑定
- MinIO 或本地对象存储
- 文件元数据入库

验收：

- 每个展项可绑定多份资料
- 文件可预览或下载
- 文件资料能作为后续 RAG 的来源

### 阶段 5：Excel 导入

目标：降低历史资料录入成本。

交付：

- Excel 模板
- 上传 Excel
- 字段映射
- 导入预览
- 错误行提示
- 批量写入展项与图谱关系

验收：

- 可通过 Excel 一次导入多条展项
- 材料、主题、供应商、业主等关系能自动入库
- 导入失败时能指出具体行和字段
- 相似展项字段必须引用已有展项或同批导入展项，避免生成无效图谱关系

### 阶段 6：混合检索

目标：把当前规则语义检索升级为真实检索能力。

交付：

- pgvector 扩展
- 展项 embedding
- 文档切片 embedding
- 结构化过滤 + 关键词 + 向量检索
- 混合排序
- 匹配原因说明

示例查询：

> 找几个适合低龄儿童、预算不高、互动性强的力学展项

期望返回：

- 杠杆乐园：匹配低龄儿童、力学、机械互动、预算 20-35 万
- 滑轮挑战墙：匹配低预算、力学、多人协作、机械互动

验收：

- 检索结果能结合结构化条件和语义相似度
- 每条结果能说明主要匹配原因

### 阶段 7：RAG 智能问答

目标：基于库内资料回答问题，并返回引用来源。

交付：

- 文档解析
- 文档切片
- 向量召回
- 问答接口
- 引用来源返回
- 不确定性处理

验收：

- 问答必须基于展项档案或上传资料
- 回答中包含来源引用
- 资料中没有依据时，明确说明未找到依据

### 阶段 8：管理后台与权限

目标：支持业务人员安全维护数据。

交付：

- 登录
- 角色权限
- 管理员、编辑、只读访客
- 审核状态
- 操作日志
- 删除保护

验收：

- 不同角色能访问不同功能
- 数据变更有记录
- 重要资料不能被误删

## 6. 近期推荐开发顺序

最近 1-2 周建议优先做：

1. 提交当前前端 Demo
2. 设计 PostgreSQL 图谱数据模型
3. 搭建后端 API
4. 前端接真实接口
5. 实现图谱查询 API
6. 实现新增/编辑展项时的关系表同步

这条路线最关键，因为它把系统从“前端演示”推进到“真实数据驱动的 MVP”。

## 7. 暂不纳入 MVP 的内容

以下内容建议放到二期或更后面：

- 全自动知识图谱构建
- CAD/3D 深度解析
- 精准造价预测
- 复杂图数据库推理
- 多智能体自动生成展陈方案
- ERP/OA/合同系统深度集成
- 大规模审批流

## 8. 成功标准

MVP 成功标准：

- 业务人员能录入真实展项
- 能按结构化条件快速筛选
- 能查看展项详情和资料
- 能看到展项与项目、业主、供应商、材料、主题、相似展项的图谱关系
- 能通过自然语言检索得到可解释结果
- 能基于库内资料进行初步问答，并返回引用来源

一句话总结：

> 展项档案是数据入口，关系表是图谱基础，图谱 API 是展示和检索基础，RAG 是最后的智能层。
