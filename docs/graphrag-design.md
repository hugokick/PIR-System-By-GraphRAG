# GraphRAG 规则版设计说明

## 1. 当前目标

本阶段 GraphRAG 只做规则版最小闭环，在不修改现有 MVP 主流程的前提下，先把以下能力独立实现出来：

- 结构化过滤
- 图谱邻域扩展
- 展项与文档证据召回
- 引用来源去重与返回

本阶段不接入 pgvector、LLM、文档切片或文件解析链路。

## 2. 设计原则

### 2.1 独立模块

GraphRAG 实现放在独立模块中，不直接耦合 FastAPI 路由层。

### 2.2 先过滤后排序

显式结构化条件优先作为候选集约束，再对候选做规则打分。

### 2.3 结果可解释

每条命中结果必须能返回：

- 命中展项
- 分数
- 命中原因
- 图谱邻域上下文
- 引用来源

### 2.4 为后续升级保留接口

后续接入文档切片、pgvector 和 LLM 时，尽量复用本阶段的结果契约，而不是重写调用方式。

## 3. 当前检索流程

规则版 GraphRAG 流程如下：

1. 输入用户查询与可选结构化过滤条件
2. 对展项列表先做结构化过滤
3. 对过滤后的候选展项做规则匹配和加权评分
4. 读取候选展项的一跳图谱邻域
5. 汇总展项级和文档级证据
6. 去重后返回引用来源

## 4. 结构化过滤条件

当前支持以下过滤条件：

- `theme`
- `material`
- `interaction`
- `venue_type`
- `status`

后续如有需要，可以扩展到预算区间、项目年份和项目案例。

## 5. 规则匹配字段

当前规则打分对以下字段进行匹配：

- 展项标识：`id`、`name`
- 分类信息：`category`、`theme`、`venue_type`、`tags`
- 材料：`materials`
- 交互方式：`interactions`
- 项目侧信息：`project`、`owner`、`supplier`
- 展项说明：`description`
- 文档信息：`document.name`、`document.source_note`

## 6. 初始权重

建议初始权重如下：

- 标识匹配：`4`
- 分类匹配：`3`
- 材料匹配：`2`
- 交互匹配：`2`
- 项目侧匹配：`2`
- 文档匹配：`2`
- 说明文本匹配：`1`

本阶段权重采用规则常量，不做学习排序。

## 7. 图谱邻域扩展

当前只做一跳邻域扩展，邻居类型包括：

- 项目
- 业主
- 供应商
- 材料
- 主题
- 交互方式
- 文档
- 相似展项

一跳扩展足以提供规则版 GraphRAG 所需的图谱上下文，也便于测试和解释。

## 8. 引用来源

当前引用来源分两层：

- 展项级引用：来自展项说明
- 文档级引用：来自文档标题与来源说明

引用去重规则：

- 按 `source_type + source_id` 去重

未来扩展时，可在不改变返回结构的前提下加入：

- 文档切片引用
- 文件解析结果引用
- Excel 导入来源引用
- 向量召回来源引用

## 9. 本阶段模块边界

本阶段实现放在：

- `backend/app/graphrag/`

模块职责：

- 定义过滤模型
- 定义命中结果模型
- 执行规则版检索
- 拼装一跳邻域上下文
- 输出引用与命中原因

本阶段不做：

- 改现有 `main.py`
- 改现有 API 路由
- 改前端 API 调用
- 改数据库 schema

## 10. 与 KG 模块的关系

GraphRAG 不直接自己造图，而是优先消费 `KGSnapshot`：

- 如果外部已传入图谱快照，则直接使用
- 如果没有传入，则可根据展项列表临时构建

这样可以保证：

- 检索逻辑与构图逻辑职责分离
- 后续主线接入时只需要统一图谱来源

## 11. 后续升级路径

在文件、Excel、文档切片链路稳定后，再按顺序引入：

1. 文档切片与 chunk 级引用
2. pgvector 向量召回
3. 混合排序
4. LLM 基于证据生成答案

升级时优先替换内部召回逻辑，不轻易修改外部结果契约。

## 12. 验收标准

规则版 GraphRAG 完成后，应满足：

- 能按结构化条件过滤候选展项
- 能按规则匹配返回 Top-K 命中
- 每条命中包含原因、引用和图谱邻域
- 无命中时返回空结果而不是报错
- 整个流程可由独立单元测试覆盖
